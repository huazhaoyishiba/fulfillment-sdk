# 异常与补偿规则设计

本文档定义了 SDK 的异常处理机制和补偿规则，确保在各种异常场景下系统能够稳定运行。

## 一、异常场景分类

### 1. 找不到处理器（Handler Not Found）

**场景**：根据 `benefitType` 查找 Handler 时，未找到对应的实现。

**处理规则**：
- ✅ **错误码**：`HANDLER_NOT_FOUND`
- ✅ **是否可重试**：❌ **不可重试**（配置错误，重试也不会成功）
- ✅ **任务状态**：直接标记为 `FAILED`
- ✅ **日志记录**：记录错误日志，包含 `benefitType` 信息
- ✅ **后续处理**：需要人工介入，检查 Handler 是否已注册

**代码位置**：`FulfillmentApplicationServiceImpl.doFulfill()` 第 218-225 行

### 2. 处理器执行抛错（Handler Execution Exception）

**场景**：Handler 执行过程中抛出异常。

**处理规则**：
- ✅ **错误码**：`HANDLER_EXCEPTION`
- ✅ **是否可重试**：✅ **可重试**（可能是临时性系统错误）
- ✅ **任务状态**：
  - 如果未达最大重试次数：标记为 `RETRY_WAIT`
  - 如果已达最大重试次数：标记为 `FAILED`
- ✅ **日志记录**：记录完整异常堆栈
- ✅ **后续处理**：
  - 可重试错误：自动或手动重试
  - 不可重试错误：需要人工介入排查

**代码位置**：`FulfillmentApplicationServiceImpl.doFulfill()` 第 268-279 行

### 3. 执行超时（Handler Timeout）

**场景**：Handler 执行时间超过配置的超时时间。

**处理规则**：
- ✅ **错误码**：`HANDLER_TIMEOUT`
- ✅ **是否可重试**：✅ **可重试**（可能是网络抖动或系统负载高）
- ✅ **超时配置**：`fulfillment.handler-timeout-seconds`（默认 30 秒，0 表示不设置超时）
- ✅ **任务状态**：
  - 如果未达最大重试次数：标记为 `RETRY_WAIT`
  - 如果已达最大重试次数：标记为 `FAILED`
- ✅ **日志记录**：记录超时信息，包含执行时长
- ✅ **后续处理**：自动或手动重试

**代码位置**：`FulfillmentApplicationServiceImpl.executeHandlerWithTimeout()`

### 4. 任务失败后的处理策略

**当前实现**：
- ✅ **手动重试**：通过 `retryByTaskNo()` 接口手动触发重试
- ⚠️ **自动重试**：配置项 `fulfillment.enable-auto-retry`（默认 false）
  - 如果启用，需要配合定时任务扫描 `RETRY_WAIT` 状态的任务
  - 定时任务需要接入方自己实现

**重试规则**：
1. **可重试错误**：根据错误类型判断（见 `ErrorCode.isRetryable()`）
2. **重试次数限制**：不超过 `maxRetryCount`（默认 3 次）
3. **重试调度**：SDK 不内置调度器，由宿主应用自行调度 `retryFailedTasks()`

**代码位置**：`FulfillmentApplicationServiceImpl.markFailure()`

### 5. 补发机制（Manual Fulfill）

**场景**：用户声称未收到权益，需要人工补偿。

**处理规则**：
- ✅ **生成新任务**：补发会创建新的发放任务，不复用旧任务
- ✅ **幂等控制**：使用独立的场景码 `MANUAL_FULFILL`，防止重复补发
- ✅ **关联原任务**：可选参数 `linkOriginalTask`，如果为 true 会查询原任务（仅用于记录，不影响新任务）
- ✅ **任务编号**：新任务有独立的 `taskNo`
- ✅ **执行流程**：与正常发放流程相同，走完整的 Handler 执行链路

**与重试的区别**：
- **重试**：复用原任务，增加重试次数，保留原任务的所有日志
- **补发**：生成新任务，从 0 开始，独立的任务和日志

**代码位置**：`FulfillmentApplicationServiceImpl.manualFulfill()`

### 6. 查询不到外部订单（Query Not Found）

**场景**：按订单号查询发放状态时，未找到对应的任务。

**处理规则**：
- ✅ **按订单号查询**（`queryByOrderNo`）：
  - 返回 `null`（不抛异常）
  - 原因：订单可能还未支付，或还未调用发放接口
  - 由调用方决定如何处理
- ✅ **按任务号查询**（`queryByTaskNo`）：
  - 抛出 `BizException`，错误码 `TASK_NOT_FOUND`
  - 原因：任务号是 SDK 内部生成的，不存在说明数据异常

**代码位置**：
- `queryByOrderNo()`：第 393-409 行
- `queryByTaskNo()`：第 411-435 行

## 二、错误码分类

### 可重试错误（临时性错误）

| 错误码 | 说明 | 典型场景 |
|--------|------|---------|
| `HANDLER_TIMEOUT` | Handler 执行超时 | 网络抖动、系统负载高 |
| `HANDLER_EXCEPTION` | Handler 执行异常 | 系统临时错误 |
| `NETWORK_ERROR` | 网络错误 | 网络临时不可用 |
| `DATABASE_ERROR` | 数据库连接错误 | 数据库临时不可用 |
| `DOWNSTREAM_UNAVAILABLE` | 下游服务不可用 | 下游服务临时不可用 |

**处理策略**：自动或手动重试，直到成功或达到最大重试次数。

### 不可重试错误（永久性错误）

| 错误码 | 说明 | 典型场景 |
|--------|------|---------|
| `INVALID_BENEFIT_TYPE` | 无效的权益类型 | 配置错误 |
| `HANDLER_NOT_FOUND` | 未找到 Handler | 缺少实现 |
| `HANDLER_BUSINESS_FAILURE` | Handler 业务失败 | 业务逻辑拒绝 |
| `VALIDATION_ERROR` | 参数校验失败 | 参数错误 |
| `DATA_INCONSISTENCY` | 数据不一致 | 数据问题 |

**处理策略**：直接标记失败，需要人工介入排查和修复。

## 三、补偿机制设计

### 1. 自动重试（可选）

**配置**：
```yaml
fulfillment:
  default-max-retry-count: 3      # 最大重试次数
  retry-batch-size: 50            # 每次最多处理的任务数
  # 注意：SDK 不内置调度器，需宿主自行调度 retryFailedTasks()
```

**实现方式**：
- SDK 不包含定时任务实现
- 接入方需要自己实现定时任务，扫描 `RETRY_WAIT` 状态的任务
- 定时任务调用 `retryByTaskNo()` 接口

**示例定时任务**：
```java
@Scheduled(fixedDelay = 60000) // 每分钟执行一次
public void autoRetryFailedTasks() {
    // 查询 RETRY_WAIT 状态的任务
    // 调用 retryByTaskNo()
}
```

### 2. 手动重试

**接口**：`GET /api/fulfillment/retry/{taskNo}`

**规则**：
- ✅ 只能重试失败的任务（`RETRY_WAIT` 或 `FAILED`）
- ✅ 不能重试已成功的任务
- ✅ 不能重试已达最大重试次数的任务
- ✅ 重试会复用原任务，增加重试次数

### 3. 手动补发

**接口**：`POST /api/fulfillment/manual-fulfill`

**规则**：
- ✅ 生成新的发放任务
- ✅ 独立的幂等控制（场景码：`MANUAL_FULFILL`）
- ✅ 可选关联原任务（仅用于记录）
- ✅ 完整的执行流程和日志记录

**使用场景**：
- 用户投诉未收到权益
- 运营活动补偿
- 系统故障后的补偿

## 四、异常处理流程图

```
支付确认请求
    ↓
幂等校验
    ↓
创建任务
    ↓
查找 Handler
    ├─ 找不到 → HANDLER_NOT_FOUND → FAILED（不可重试）
    └─ 找到 → 执行 Handler（带超时控制）
            ├─ 超时 → HANDLER_TIMEOUT → RETRY_WAIT（可重试）
            ├─ 异常 → HANDLER_EXCEPTION → RETRY_WAIT（可重试）
            ├─ 业务失败 → HANDLER_BUSINESS_FAILURE → FAILED（不可重试）
            └─ 成功 → SUCCESS
    ↓
判断是否可重试
    ├─ 可重试 + 未达上限 → RETRY_WAIT（等待重试）
    └─ 不可重试 或 已达上限 → FAILED（最终失败）
```

## 五、配置说明

```yaml
fulfillment:
  # 默认最大重试次数
  default-max-retry-count: 3
  
  # Handler 执行超时时间（秒），0 表示不设置超时
  handler-timeout-seconds: 30
  
  # 是否启用自动重试
  # 注意：SDK 不内置调度器，由宿主自行调度 retryFailedTasks()
  
  # 幂等记录过期时间（小时）
  idempotent-expire-hours: 24
  
  # 错误信息最大长度
  max-error-message-length: 500
  
  # 结果摘要最大长度
  max-result-summary-length: 500
```

## 六、最佳实践

### 1. Handler 实现建议

- ✅ **快速失败**：对于不可恢复的错误，应该快速返回失败，不要重试
- ✅ **幂等性**：Handler 实现应该是幂等的，支持重复执行
- ✅ **超时控制**：Handler 内部也应该有超时控制，避免长时间阻塞
- ✅ **错误码规范**：返回明确的错误码，便于 SDK 判断是否可重试

### 2. 监控告警

建议接入方实现以下监控：
- ✅ 失败任务数量（按错误码分类）
- ✅ 重试任务数量
- ✅ Handler 执行时长（P50、P95、P99）
- ✅ 超时任务数量
- ✅ 补发任务数量

### 3. 人工介入场景

以下场景需要人工介入：
- ❌ `HANDLER_NOT_FOUND`：检查 Handler 是否已注册
- ❌ `INVALID_BENEFIT_TYPE`：检查配置是否正确
- ❌ `VALIDATION_ERROR`：检查参数是否正确
- ❌ `DATA_INCONSISTENCY`：检查数据是否一致
- ❌ 达到最大重试次数仍失败：需要排查根本原因

## 七、总结

SDK 提供了完善的异常处理和补偿机制：

1. ✅ **异常分类**：区分可重试和不可重试错误
2. ✅ **超时控制**：防止 Handler 长时间阻塞
3. ✅ **重试机制**：支持手动和自动重试
4. ✅ **补发机制**：支持人工补偿，生成新任务
5. ✅ **查询处理**：区分订单号和任务号查询的不同处理
6. ✅ **日志记录**：完整的异常日志，便于排查

接入方需要：
- 实现 Handler 时遵循最佳实践
- 根据业务需求配置重试策略
- 实现监控告警
- 处理需要人工介入的场景
