## 1. SDK 是什么 / 不是什么

### ✅ SDK 负责

| 能力 | 说明 |
|------|------|
| 接收支付成功事件 | 外部系统确认支付后调用 SDK |
| 幂等校验 | 同一请求多次调用只执行一次 |
| 创建发放任务 | 记录任务状态，支持追溯 |
| Handler 调度 | 根据权益类型自动匹配处理器 |
| 超时控制 | Handler 执行超时自动中断 |
| 执行日志 | 每次尝试都留底，便于排障 |
| 失败重试 | 自动/手动重试失败任务 |
| 手动补发 | 人工补偿场景 |
| 状态查询 | 按订单号/任务号查询 |

### ❌ SDK 不负责

- 不管理用户表、订单主表、会员表、积分表等业务数据
- 不绑定具体数据库驱动（由你的项目提供）
- 不绑定特定 ORM（默认提供 MyBatis 实现，可替换）
- 不包含 Web 层（无 Controller，你自行决定如何暴露接口）

**一句话总结**：SDK 管「这次发放有没有成功」，你管「成功后把权益落到哪里」。

## 2. 整体架构

```
你的业务系统
│
├── Controller / MQ Consumer          ← 你自己写，接收支付回调
│       │
│       ▼
│  FulfillmentApplicationService      ← SDK 核心入口（注入即用）
│       │
│       ├── 幂等校验
│       ├── 创建支付记录 + 发放任务
│       ├── Handler 调度 ──────────→  你实现的 FulfillmentHandler
│       │                              ├── MembershipHandler
│       │                              ├── CouponHandler
│       │                              └── YourCustomHandler
│       ├── 记录执行日志
│       └── 更新任务状态（成功/重试/失败）
│
├── 数据库（你的项目提供）
│       ├── sdk_payment_record         ← SDK 的 4 张表
│       ├── sdk_idempotent_record
│       ├── sdk_fulfillment_task
│       └── sdk_fulfillment_log
│
└── application.yml（你配置数据源 + SDK 参数）
```

## 3. 快速开始（5 分钟跑通）

以一个 Spring Boot + MySQL + MyBatis 项目为例，最快让 SDK 跑起来。

### 第 1 步：加依赖

在你项目的 `pom.xml` 中加入：

```xml
<!-- 1. 引入 SDK Starter -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>fulfillment-spring-boot-starter-boot2</artifactId>
    <version>0.0.1-SNAPSHOT</version>
</dependency>

<!-- 2. 引入 SDK 的 MyBatis 默认存储实现 -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>fulfillment-storage-mybatis</artifactId>
    <version>0.0.1-SNAPSHOT</version>
</dependency>

<!-- 3. 必须：MyBatis（SDK 的默认 Repository 实现依赖它） -->
<dependency>
    <groupId>org.mybatis.spring.boot</groupId>
    <artifactId>mybatis-spring-boot-starter</artifactId>
    <version>2.3.1</version>
</dependency>

<!-- 4. 必须：你项目的数据库驱动 -->
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
</dependency>
```

> **注意**：SDK 采用多模块架构：
> - `fulfillment-spring-boot-starter-boot2`：Starter 只装配内核能力，不默认带 JDBC/MyBatis
> - `fulfillment-core`：核心逻辑（自动传递）
> - `fulfillment-storage-spi`：持久化接口（自动传递）
> - `fulfillment-storage-mybatis`：MyBatis 实现（**需显式引入**，可选）

### 第 2 步：建表

在你的 MySQL 数据库中执行 SDK 提供的建表脚本：

```
fulfillment-storage-mybatis/src/main/resources/schema-mysql.sql
```

> 其他数据库请使用对应的脚本：`schema-postgresql.sql`、`schema-oracle.sql`、`schema-dm.sql`、`schema-kingbase.sql`
> 脚本均位于 `fulfillment-storage-mybatis/src/main/resources/` 目录下。

### 第 3 步：加配置

在你项目的 `application.yml` 中：

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/your_db?useSSL=false&characterEncoding=utf-8
    username: root
    password: 123456
    driver-class-name: com.mysql.cj.jdbc.Driver

# MyBatis 驼峰映射（SDK 的 MyBatis 实现需要）
mybatis:
  configuration:
    map-underscore-to-camel-case: true

# SDK 配置（以下为默认值，可不写）
fulfillment:
  default-max-retry-count: 3
  handler-timeout-seconds: 30
```

### 第 4 步：写一个 Handler

```java
package com.yourcompany.handler;

import com.example.fulfillment.domain.entity.FulfillmentTask;
import com.example.fulfillment.service.handler.HandlerResult;
import com.example.fulfillment.spi.FulfillmentHandler;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

@Component
public class MembershipHandler implements FulfillmentHandler {

    @Autowired
    private YourMembershipService membershipService; // 你自己的业务 Service

    @Override
    public boolean supports(String benefitTypeCode) {
        return "MEMBERSHIP".equals(benefitTypeCode);
    }

    @Override
    public HandlerResult fulfill(FulfillmentTask task) {
        // 1. 从快照中解析权益配置
        String config = task.getBenefitConfigSnapshot();
        // 例如: {"days": 30, "level": "VIP"}
        int days = parseFromJson(config, "days");

        // 2. 调用你自己的业务逻辑
        membershipService.extendMembership(task.getBizUserRef(), days);

        // 3. 返回成功
        return HandlerResult.success("会员延长" + days + "天");
    }
}
```

### 第 5 步：在你的 Controller / MQ 里调用 SDK

**重要**：SDK **不提供 Controller**，只提供 `FulfillmentApplicationService` 接口。你需要自己写 Controller 来暴露 HTTP 接口。

```java
@RestController
@RequestMapping("/payment")
public class PaymentCallbackController {

    @Autowired
    private FulfillmentApplicationService fulfillmentService;  // SDK 提供的 Service

    @PostMapping("/callback")
    public ResponseEntity<?> onPaymentSuccess(@RequestBody PaymentNotify notify) {
        ConfirmPaymentRequest request = new ConfirmPaymentRequest();
        request.setBizOrderNo(notify.getOrderNo());
        request.setBizUserRef(notify.getUserId());
        request.setPaymentNo(notify.getTransactionId());
        request.setChannelCode("WECHAT");
        request.setPaidAmount(notify.getAmount());
        request.setBenefitTypeCode("MEMBERSHIP");
        request.setBenefitConfigSnapshot("{\"days\":30,\"level\":\"VIP\"}");

        FulfillmentResponse result = fulfillmentService.confirmPaidAndFulfill(request);

        if (result.isSuccess()) {
            return ResponseEntity.ok("发放成功: " + result.getTaskNo());
        } else {
            return ResponseEntity.ok("发放失败: " + result.getErrorCode());
        }
    }
}
```

**完成！** 此时支付回调进来后，SDK 会自动完成幂等 → 建任务 → 调度你的 Handler → 记录日志 → 返回结果。

### 4.4 实现权益 Handler

**这是你唯一需要写的核心代码。** 每种权益类型对应一个 Handler。

#### Handler 开发规范

```java
import com.example.fulfillment.domain.entity.FulfillmentTask;
import com.example.fulfillment.service.handler.HandlerResult;
import com.example.fulfillment.spi.FulfillmentHandler;
import org.springframework.stereotype.Component;

@Component  // 必须是 Spring Bean
public class YourHandler implements FulfillmentHandler {

    @Override
    public boolean supports(String benefitTypeCode) {
        // 声明此 Handler 处理哪种权益类型
        return "MEMBERSHIP".equals(benefitTypeCode);
    }

    @Override
    public HandlerResult fulfill(FulfillmentTask task) {
        // task 中可获取的信息：
        // task.getBizOrderNo()            - 业务订单号
        // task.getBizUserRef()            - 用户标识（你传入的）
        // task.getBenefitTypeCode()       - 权益类型编码（字符串）
        // task.getBenefitConfigSnapshot() - 权益配置快照（JSON 字符串）
        // task.getRetryCount()            - 当前第几次重试

        try {
            // 在这里写你的业务逻辑
            // ...

            // 成功：返回摘要（会记入日志）
            return HandlerResult.success("处理完成，详情: xxx");

        } catch (SomeRetryableException e) {
            // 可重试的失败（如网络抖动、下游超时）
            // 使用 SDK 预定义的可重试错误码
            return HandlerResult.fail("DOWNSTREAM_UNAVAILABLE", "下游服务暂时不可用: " + e.getMessage());

        } catch (SomeBusinessException e) {
            // 不可重试的业务失败（如余额不足、参数错误）
            return HandlerResult.fail("HANDLER_BUSINESS_FAILURE", "业务拒绝: " + e.getMessage());
        }
        // 注意：如果抛出未捕获的异常，SDK 会自动捕获，标记为可重试（HANDLER_EXCEPTION）
    }
}
```

#### Handler 返回值说明

| 方法 | 含义 |
|------|------|
| `HandlerResult.success("摘要")` | 发放成功，任务标记为 `SUCCESS` |
| `HandlerResult.fail("错误码", "错误信息")` | 发放失败，SDK 根据错误码决定是否重试 |

#### 权益类型编码（benefitTypeCode）

SDK 使用 **字符串** 作为权益类型编码（`benefitTypeCode`），不限于固定枚举，支持任意自定义类型。

**常用内置编码（建议约定）**：

| 编码 | 说明 |
|------|------|
| `"MEMBERSHIP"` | 会员 |
| `"FILE_ACCESS"` | 文件下载权 |
| `"API_QUOTA"` | API 调用次数 |
| `"LICENSE_KEY"` | 卡密/激活码 |
| `"COUPON"` | 优惠券 |
| `"POINTS"` | 积分 |

你可以使用任意字符串，如 `"VIP_UPGRADE"`、`"TRIAL_30D"` 等，只要你的 Handler 能匹配即可。

#### 多个 Handler 示例

```java
// 会员 Handler
@Component
public class MembershipHandler implements FulfillmentHandler {
    public boolean supports(String benefitTypeCode) { return "MEMBERSHIP".equals(benefitTypeCode); }
    public HandlerResult fulfill(FulfillmentTask task) {
        // 延长会员天数
        return HandlerResult.success("会员已开通30天");
    }
}

// 卡密 Handler
@Component
public class LicenseKeyHandler implements FulfillmentHandler {
    public boolean supports(String benefitTypeCode) { return "LICENSE_KEY".equals(benefitTypeCode); }
    public HandlerResult fulfill(FulfillmentTask task) {
        String key = licenseService.allocateKey();
        return HandlerResult.success("已分配卡密: " + key);
    }
}

// 优惠券 Handler（自定义权益类型）
@Component
public class CouponHandler implements FulfillmentHandler {
    public boolean supports(String benefitTypeCode) { return "COUPON".equals(benefitTypeCode); }
    public HandlerResult fulfill(FulfillmentTask task) {
        String config = task.getBenefitConfigSnapshot();
        couponService.issue(task.getBizUserRef(), config);
        return HandlerResult.success("优惠券已发放");
    }
}
```

## 5. 核心 API 参考

### 5.1 confirmPaidAndFulfill — 支付确认并发放

**这是最核心的方法。** 支付成功后调用，SDK 完成幂等 → 建任务 → 调度 Handler → 记录结果。

```java
ConfirmPaymentRequest request = new ConfirmPaymentRequest();
request.setBizOrderNo("ORD-20260227-001");      // 必填：你的业务订单号
request.setBizUserRef("USER-10086");              // 必填：用户标识（你定义的）
request.setPaymentNo("PAY-WX-123456789");         // 必填：支付流水号
request.setChannelCode("WECHAT");                 // 必填：支付渠道编码
request.setPaidAmount(new BigDecimal("99.00"));    // 必填：支付金额
request.setBenefitTypeCode("MEMBERSHIP");          // 必填：权益类型编码（字符串）
request.setBenefitConfigSnapshot(                  // 必填：权益配置快照（JSON）
    "{\"days\":30,\"level\":\"VIP\"}"
);
request.setIdempotentKey("MY-CUSTOM-KEY");         // 可选：自定义幂等键（不传则自动生成）

FulfillmentResponse result = fulfillmentService.confirmPaidAndFulfill(request);
```

**参数说明：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `bizOrderNo` | String | ✅ | 你的业务订单号 |
| `bizUserRef` | String | ✅ | 用户业务标识（你定义，如 userId、openId） |
| `paymentNo` | String | ✅ | 支付流水号（微信/支付宝返回的交易号） |
| `channelCode` | String | ✅ | 支付渠道编码（WECHAT / ALIPAY / BANK 等） |
| `paidAmount` | BigDecimal | ✅ | 支付金额 |
| `benefitTypeCode` | String | ✅ | 权益类型编码（任意字符串，如 `"MEMBERSHIP"`、`"COUPON"` 等） |
| `benefitConfigSnapshot` | String | ✅ | 权益配置快照（JSON），用于 Handler 解析和重试时保持一致 |
| `idempotentKey` | String | 否 | 自定义幂等键。不传则 SDK 自动根据 (场景+订单号+支付号) 生成 |

**返回值 `FulfillmentResponse`：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `success` | boolean | 是否成功 |
| `taskNo` | String | SDK 生成的任务编号（唯一标识） |
| `status` | String | 任务状态：`SUCCESS` / `RETRY_WAIT` / `FAILED` |
| `errorCode` | String | 错误码（仅失败时有值） |
| `message` | String | 结果摘要 / 错误信息 |

**幂等行为：** 同一幂等键重复调用不会重复执行，直接返回上次结果（`message` 为 `"idempotent replay"`）。

### 5.2 manualFulfill — 手动补发

用于售后补偿场景，**生成新的发放任务**（不复用原任务）。

```java
ManualFulfillRequest request = new ManualFulfillRequest();
request.setIdempotentKey("MANUAL-ORD001-20260227");  // 必填：幂等键
request.setBizOrderNo("ORD-20260227-001");            // 必填
request.setBizUserRef("USER-10086");                   // 必填
request.setBenefitTypeCode("MEMBERSHIP");               // 必填：权益类型编码（字符串）
request.setBenefitConfigSnapshot("{\"days\":30}");      // 必填
request.setReason("用户投诉未到账，人工补发");             // 建议填写，用于审计

FulfillmentResponse result = fulfillmentService.manualFulfill(request);
```

> **补发 vs 重试的区别**：
> - **重试**：针对失败任务，复用原任务 ID，增加重试次数
> - **补发**：生成全新任务，独立执行，独立幂等

### 5.3 retryByTaskNo — 手动重试

用于管理后台，对单个失败任务手动触发重试。

```java
try {
    FulfillmentResponse result = fulfillmentService.retryByTaskNo("FT-abc123def456");
    // 重试结果
} catch (BizException e) {
    switch (e.getCode()) {
        case "TASK_NOT_FOUND":          // 任务不存在
        case "TASK_ALREADY_SUCCESS":    // 已成功，不需要重试
        case "TASK_PROCESSING":         // 正在处理中，稍后再试
        case "MAX_RETRY_EXCEEDED":      // 已达最大重试次数
        case "TASK_STATUS_NOT_RETRYABLE": // 状态不允许重试
        case "CONCURRENT_MODIFICATION":  // 并发冲突，其他人正在重试
    }
}
```

## 7. 错误码与重试规则

SDK 通过错误码决定失败后是否可以自动重试：

### 可重试的错误码（临时性故障，自动重试）

| 错误码 | 说明 | 典型场景 |
|--------|------|---------|
| `HANDLER_TIMEOUT` | Handler 执行超时 | 网络抖动、系统负载高 |
| `HANDLER_EXCEPTION` | Handler 抛出未捕获异常 | 临时性系统错误 |
| `NETWORK_ERROR` | 网络错误 | 网络临时不可用 |
| `DATABASE_ERROR` | 数据库连接错误 | 数据库临时不可用 |
| `DOWNSTREAM_UNAVAILABLE` | 下游服务不可用 | 第三方接口暂时挂了 |

### 不可重试的错误码（永久性错误，直接标记失败）

| 错误码 | 说明 | 典型场景 |
|--------|------|---------|
| `HANDLER_NOT_FOUND` | 未找到对应 Handler | 忘记实现 Handler |
| `INVALID_BENEFIT_TYPE` | 无效的权益类型 | 配置错误 |
| `HANDLER_BUSINESS_FAILURE` | Handler 返回业务失败 | 余额不足、库存不够 |
| `VALIDATION_ERROR` | 参数校验失败 | 参数错误 |
| `DATA_INCONSISTENCY` | 数据不一致 | 需要人工介入 |

### 在 Handler 中使用错误码

```java
// 可重试的失败 → 任务状态变为 RETRY_WAIT，等待下次重试
return HandlerResult.fail("DOWNSTREAM_UNAVAILABLE", "第三方接口 500 错误");

// 不可重试的失败 → 任务状态直接变为 FAILED
return HandlerResult.fail("HANDLER_BUSINESS_FAILURE", "用户已被冻结，无法发放");
```

### 任务状态流转

```
PROCESSING ──成功──→ SUCCESS
     │
     │──失败（可重试 且 未超限）──→ RETRY_WAIT ──重试──→ PROCESSING
     │
     │──失败（不可重试 或 已超限）──→ FAILED
```

## 8. 自动重试配置

SDK 只暴露调度入口方法（`retryFailedTasks()`、`recoverStuckTasks()`），**不内置调度器**。由宿主应用自行决定调度方式：

### 方式 A：Spring @Scheduled（适合单实例）

```java
@Component
@EnableScheduling
public class FulfillmentRetryTask {

    @Autowired
    private FulfillmentApplicationService fulfillmentService;

    @Scheduled(fixedDelay = 60000) // 每 60 秒
    public void retry() {
        int count = fulfillmentService.retryFailedTasks();
        if (count > 0) {
            log.info("本轮重试: {} 条", count);
        }
    }
}
```

### 方式 B：外部调度平台（适合分布式，推荐）

**XXL-JOB 示例：**

```java
@Component
public class FulfillmentRetryXxlJob {

    @Autowired
    private FulfillmentApplicationService fulfillmentService;

    @XxlJob("fulfillmentRetryJob")
    public void execute() {
        int count = fulfillmentService.retryFailedTasks();
        XxlJobHelper.log("本轮重试: {} 条", count);
        XxlJobHelper.handleSuccess("重试完成: " + count + " 条");
    }
}
```

**Quartz 示例：**

```java
@Component
public class FulfillmentRetryQuartzJob extends QuartzJobBean {

    @Autowired
    private FulfillmentApplicationService fulfillmentService;

    @Override
    protected void executeInternal(JobExecutionContext context) {
        int count = fulfillmentService.retryFailedTasks();
        log.info("Quartz 自动重试: {} 条", count);
    }
}
```

**Spring 自带 @Scheduled（简单项目也可以自己写）：**

```java
@Component
@EnableScheduling
public class FulfillmentRetryJob {

    @Autowired
    private FulfillmentApplicationService fulfillmentService;

    @Scheduled(fixedDelay = 30000) // 每 30 秒
    public void retry() {
        fulfillmentService.retryFailedTasks();
    }
}
```

## 9. 自定义 Repository 实现（可选）

如果你不想使用 MyBatis，可以自行实现 SDK 的 4 个 Repository 接口。

### 需要实现的接口

| 接口 | 说明 |
|------|------|
| `PaymentRecordRepository` | 支付记录的增查 |
| `IdempotentRecordRepository` | 幂等记录的增查改 |
| `FulfillmentTaskRepository` | 发放任务的增查改（含乐观锁） |
| `FulfillmentLogRepository` | 发放日志的增查 |

### 实现示例（以 JdbcTemplate 为例）

```java
@Repository
public class JdbcPaymentRecordRepository implements PaymentRecordRepository {

    private final JdbcTemplate jdbcTemplate;

    public JdbcPaymentRecordRepository(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }

    @Override
    public PaymentRecord findByPaymentNo(String paymentNo) {
        String sql = "SELECT * FROM sdk_payment_record WHERE payment_no = ?";
        List<PaymentRecord> list = jdbcTemplate.query(sql, new BeanPropertyRowMapper<>(PaymentRecord.class), paymentNo);
        return list.isEmpty() ? null : list.get(0);
    }

    @Override
    public void save(PaymentRecord record) {
        String sql = "INSERT INTO sdk_payment_record(biz_order_no, payment_no, channel_code, paid_amount, status) VALUES(?,?,?,?,?)";
        jdbcTemplate.update(sql, record.getBizOrderNo(), record.getPaymentNo(),
                record.getChannelCode(), record.getPaidAmount(), record.getStatus());
    }
}
```

### 特别注意：FulfillmentTaskRepository.updateStatusWithVersion

此方法**必须实现乐观锁**，SQL 中必须包含 `WHERE id = ? AND version = ?`，更新时 `version = version + 1`：

```java
@Override
public int updateStatusWithVersion(FulfillmentTask task) {
    String sql = "UPDATE sdk_fulfillment_task "
        + "SET status = ?, retry_count = ?, last_error_code = ?, last_error_msg = ?, "
        + "version = version + 1, updated_at = now() "
        + "WHERE id = ? AND version = ?";
    return jdbcTemplate.update(sql,
            task.getStatus(), task.getRetryCount(), task.getLastErrorCode(), task.getLastErrorMsg(),
            task.getId(), task.getVersion());
}
```

### 特别注意：FulfillmentTaskRepository.findRetryableTasks

此方法用于批量自动重试，查询可重试任务：

```java
@Override
public List<FulfillmentTask> findRetryableTasks(int limit) {
    String sql = "SELECT * FROM sdk_fulfillment_task "
        + "WHERE status = 'RETRY_WAIT' AND retry_count < max_retry_count "
        + "ORDER BY updated_at ASC "
        + "LIMIT ?";
    return jdbcTemplate.query(sql, new BeanPropertyRowMapper<>(FulfillmentTask.class), limit);
}
```

> 只要你注册了自己的 `@Repository` Bean，SDK 的 MyBatis 默认实现会自动让位（`@ConditionalOnMissingBean`）。

### 9.2 使用 MyBatis-Plus 实现（推荐）

如果你的项目使用 **MyBatis-Plus**，建议**不引入** `fulfillment-storage-mybatis` 模块，而是自行实现 Repository 接口。

**原因**：
- SDK 的 MyBatis 实现依赖 `mybatis-spring-boot-starter`，可能与 MyBatis-Plus 冲突
- SDK 的 Mapper 使用原生 MyBatis 注解，与 MyBatis-Plus 的 `BaseMapper` 不兼容
- 自行实现可以充分利用 MyBatis-Plus 的特性（如 `LambdaQueryWrapper`、`UpdateWrapper` 等）

**实现示例**：

```java
package com.yourcompany.fulfillment.repository;

import com.baomidou.mybatisplus.core.conditions.query.LambdaQueryWrapper;
import com.baomidou.mybatisplus.core.conditions.update.LambdaUpdateWrapper;
import com.baomidou.mybatisplus.extension.service.impl.ServiceImpl;
import com.example.fulfillment.core.repository.PaymentRecordRepository;
import com.example.fulfillment.domain.entity.PaymentRecord;
import org.springframework.stereotype.Repository;

@Repository
public class MyBatisPlusPaymentRecordRepository extends ServiceImpl<PaymentRecordMapper, PaymentRecord> 
        implements PaymentRecordRepository {

    @Override
    public PaymentRecord findByPaymentNo(String paymentNo) {
        return this.getOne(new LambdaQueryWrapper<PaymentRecord>()
                .eq(PaymentRecord::getPaymentNo, paymentNo)
                .last("LIMIT 1"));
    }

    @Override
    public void save(PaymentRecord record) {
        this.save(record);
    }

    @Override
    public void updateStatus(PaymentRecord record) {
        this.update(new LambdaUpdateWrapper<PaymentRecord>()
                .eq(PaymentRecord::getId, record.getId())
                .set(PaymentRecord::getStatus, record.getStatus()));
    }
}
```

**注意事项**：
1. **必须实现乐观锁**：`FulfillmentTaskRepository.updateStatusWithVersion()` 必须使用 `version` 字段实现乐观锁
2. **必须实现批量查询**：`FulfillmentTaskRepository.findRetryableTasks()` 用于自动重试
3. **确保事务**：所有 Repository 方法应在事务中执行

**依赖配置**：

```xml
<!-- 只引入 Starter（不引入 MyBatis 实现） -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>fulfillment-spring-boot-starter-boot2</artifactId>
    <version>0.0.1-SNAPSHOT</version>
</dependency>

<!-- 你的 MyBatis-Plus 依赖 -->
<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>mybatis-plus-boot-starter</artifactId>
    <version>3.5.3.1</version>
</dependency>
```

> **说明**：Starter 中 `fulfillment-storage-mybatis` 已标记为 `optional=true`，不会自动传递。只要你不显式引入 `fulfillment-storage-mybatis`，就不会引入 SDK 的 MyBatis 实现。

## 11. 常见问题 FAQ

### Q1: 启动报 `PaymentRecordRepository that could not be found`

**原因**：没有可用的 Repository 实现。Starter 本身不默认带 JDBC / MyBatis，需要你显式引入存储实现。

**解决**：
- **方式 A**：显式引入 `fulfillment-storage-mybatis` 和 `mybatis-spring-boot-starter` 依赖（见 [4.1 节](#41-引入依赖)），使用 SDK 的 MyBatis 实现
- **方式 B**：自行实现 Repository 接口（见 [9.2 节](#92-使用-mybatis-plus-实现推荐)），适合 MyBatis-Plus 项目

> **装配条件**：SDK 的完整发放服务（`FulfillmentApplicationService`）只在 4 个 Repository Bean 全部存在时才自动装配。如果只引入 Starter 不引入存储实现，项目能正常启动但不会注册完整发放服务。

### Q1.1: 项目使用 MyBatis-Plus，SDK 的 MyBatis 实现会冲突吗？

**答**：**可能会冲突**。建议：
1. **不引入** `fulfillment-storage-mybatis` 模块
2. **自行实现** Repository 接口，使用 MyBatis-Plus 的方式（见 [9.2 节](#92-使用-mybatis-plus-实现推荐)）
3. SDK 的自动装配会检测到你已注册的 Repository Bean，不会加载 MyBatis 实现（`@ConditionalOnMissingBean`）

**为什么可能冲突**：
- SDK 的 MyBatis 实现依赖 `mybatis-spring-boot-starter`
- SDK 使用 `@MapperScan` 扫描 Mapper，可能与 MyBatis-Plus 的扫描冲突
- SDK 的 Mapper 使用原生 MyBatis 注解，与 MyBatis-Plus 的 `BaseMapper` 不兼容

---

### Q2: 同一个支付回调被多次调用会怎样？

**答**：SDK 内置幂等机制。第一次调用正常执行，后续相同幂等键的调用直接返回第一次的结果，不会重复执行 Handler。

返回的 `FulfillmentResponse.message` 为 `"idempotent replay"`。

---

### Q3: Handler 执行超时会怎样？

**答**：SDK 会对执行线程发送中断信号，任务标记为 `RETRY_WAIT`（可重试），错误码为 `HANDLER_TIMEOUT`。超时时间通过 `fulfillment.handler-timeout-seconds` 配置。

> 注意：中断是协作式机制。若 Handler 使用不可中断阻塞调用，或未检查中断状态，超时后业务逻辑仍可能继续运行。建议在 Handler 中显式处理 `InterruptedException`，并在循环/长耗时逻辑中检查 `Thread.currentThread().isInterrupted()`。

---

### Q4: Handler 抛出异常会怎样？

**答**：SDK 会自动捕获异常，记录错误信息，任务标记为 `RETRY_WAIT`（可重试），错误码为 `HANDLER_EXCEPTION`。不需要在 Handler 中做全局 try-catch。

---

### Q5: 任务达到最大重试次数后怎么办？

**答**：任务状态变为 `FAILED`。此时可以：
1. 人工排查问题后调用 `manualFulfill()` 进行补发
2. 人工排查问题后调用 `retryByTaskNo()` 强制重试（如果状态是 `FAILED`）

---

### Q6: 如何自定义权益类型？

**答**：SDK 使用 **字符串** 作为权益类型编码（`benefitTypeCode`），你可以使用任意字符串。只需在 Handler 的 `supports()` 方法中匹配对应的编码即可：

```java
// 自定义权益类型 "COUPON"
@Component
public class CouponHandler implements FulfillmentHandler {
    @Override
    public boolean supports(String benefitTypeCode) {
        return "COUPON".equals(benefitTypeCode);
    }

    @Override
    public HandlerResult fulfill(FulfillmentTask task) {
        // 解析配置并发放优惠券
        String config = task.getBenefitConfigSnapshot();
        couponService.issue(task.getBizUserRef(), config);
        return HandlerResult.success("优惠券已发放");
    }
}
```

如果需要用一个 Handler 处理多种子类型，可以在 `benefitConfigSnapshot` 中包含子类型信息：

```java
@Override
public HandlerResult fulfill(FulfillmentTask task) {
    JSONObject config = JSON.parseObject(task.getBenefitConfigSnapshot());
    String subType = config.getString("subType");

    if ("COUPON".equals(subType)) {
        return issueCoupon(task);
    } else if ("POINTS".equals(subType)) {
        return addPoints(task);
    } else {
        return HandlerResult.fail("HANDLER_BUSINESS_FAILURE", "未知子类型: " + subType);
    }
}
```

---

### Q7: benefitConfigSnapshot 为什么是快照？

**答**：因为用户的权益配置可能随时变化（比如今天会员 30 天，明天改成 15 天）。快照保证：
- 重试时使用的是**下单时**的配置，而不是当前最新配置
- 审计时能追溯当时的发放条件

---

### Q8: 多个实例部署时，自动重试会重复执行吗？

**答**：不会。SDK 使用乐观锁（`version` 字段）确保同一任务只被一个实例处理。但建议分布式场景使用外部调度平台（如 XXL-JOB），由调度平台保证只有一个实例触发任务。

---

### Q9: SDK 会不会和我项目的事务冲突？

**答**：不会。SDK 的事务边界设计为：
- **阶段 1**（事务内）：幂等检查 + 创建支付记录 + 创建任务（毫秒级，极快）
- **阶段 2**（事务外）：执行 Handler（可能耗时较长，不会占用数据库连接）

Handler 如果需要自己的事务，可以在 Handler 内部使用 `@Transactional`。

---

### Q10: SDK 不含 Controller，我怎么对外暴露接口？

**答**：SDK **只提供 Service 接口**（`FulfillmentApplicationService`），不提供 Controller。你在自己的项目中写 Controller，调用 SDK 的 Service：

```java
@RestController
@RequestMapping("/api/fulfillment")
public class YourFulfillmentController {

    @Autowired
    private FulfillmentApplicationService fulfillmentService;  // SDK 提供的 Service

    @PostMapping("/retry/{taskNo}")
    public FulfillmentResponse retry(@PathVariable String taskNo) {
        return fulfillmentService.retryByTaskNo(taskNo);
    }

    @GetMapping("/query/{orderNo}")
    public FulfillmentStatusResponse query(@PathVariable String orderNo) {
        return fulfillmentService.queryByOrderNo(orderNo);
    }
}
```

这样设计的好处：
- SDK 不绑定 Web 框架（你可以用 Spring MVC、WebFlux、或者直接 MQ Consumer）
- 你可以自由设计 API 路径、参数格式、响应格式
- SDK 专注于业务逻辑，不涉及 HTTP 协议细节

---

### Q11: 只引入 Starter 不引入存储实现，项目能启动吗？

**答**：**能**。Starter 默认不要求 DataSource / JDBC / MyBatis。只引入 Starter 时：
- 项目能正常启动
- SDK 只装配基础内核能力（`FulfillmentProperties`、`FulfillmentHandlerRegistry`）
- 不会注册完整发放服务（`FulfillmentApplicationService`）
- 控制台会提示：未发现 Repository 实现，请引入存储模块

需要完整功能时，再引入 `fulfillment-storage-mybatis`（或自行实现 Repository）即可。

```java
 * 支付回调 Controller（你自己实现）
```

### MembershipHandler.java

```java
@Component
public class MembershipHandler implements FulfillmentHandler {

    @Autowired
    private MembershipService membershipService;

    @Override
    public boolean supports(String benefitTypeCode) {
        return "MEMBERSHIP".equals(benefitTypeCode);
    }

    @Override
    public HandlerResult fulfill(FulfillmentTask task) {
        JSONObject config = JSON.parseObject(task.getBenefitConfigSnapshot());
        int days = config.getIntValue("days");
        String level = config.getString("level");

        try {
            membershipService.activate(task.getBizUserRef(), days, level);
            return HandlerResult.success("会员已开通 " + days + " 天，等级: " + level);
        } catch (MembershipAlreadyActiveException e) {
            membershipService.extend(task.getBizUserRef(), days);
            return HandlerResult.success("会员已续期 " + days + " 天");
        } catch (UserNotFoundException e) {
            return HandlerResult.fail("HANDLER_BUSINESS_FAILURE", "用户不存在: " + task.getBizUserRef());
        }
    }
}
```

## 13. SDK 模块架构说明

SDK 采用**多模块 Maven 项目**，模块职责清晰：

| 模块 | 职责 | 用户是否需要直接依赖 |
|------|------|---------------------|
| `fulfillment-core` | 核心：实体、DTO、Service 接口、Handler SPI | ❌ 自动传递 |
| `fulfillment-storage-spi` | 持久化 SPI：Repository 接口定义 | ❌ 自动传递 |
| `fulfillment-storage-mybatis` | MyBatis 实现：Mapper、Repository 实现 | ⚠️ **需显式引入**（可选） |
| `fulfillment-spring-boot-starter-boot2` | Starter：自动装配、配置、Service 实现（Spring Boot 2.x） | ✅ **必须引入** |

**接入模式**：

| 模式 | 需引入 | 效果 |
|------|--------|------|
| 轻量模式（只引内核） | `starter` | 项目能启动，只装配基础能力，不含发放服务 |
| 完整模式（默认 MyBatis） | `starter` + `storage-mybatis` + `mybatis-spring-boot-starter` | 自动装配完整发放服务 |
| 自定义存储模式 | `starter` + 自行实现 Repository | 自动装配完整发放服务（不依赖 MyBatis） |

**装配条件说明**：

| Bean | 装配条件 |
|------|---------|
| `FulfillmentProperties` | 始终装配 |
| `FulfillmentHandlerRegistry` | 始终装配 |
| `FulfillmentApplicationService` | 4 个 Repository Bean 全部存在时装配 |
| `FulfillmentTransactionService` | 4 个 Repository Bean 全部存在时装配 |
| ~~FulfillmentRetryScheduler~~ | 已移除，SDK 不内置调度器，由宿主自行调度 `retryFailedTasks()` |

**自动装配注册**：通过 `META-INF/spring.factories`（Spring Boot 2.x 标准方式）注册。
